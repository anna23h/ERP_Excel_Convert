## [扫码] 运单扫码回流：替代无货勾选页的纸质勾选 + 手工转录

- 状态: TODO
- 现象: 拣货员在打印的「拣货表+面单」上用笔勾选完成打包的订单，员工再手工把勾选结果录进 Excel，作为 stage2 有货/无货清单入口。400 单量级下逐单核对 16 位单号，错一位、跳一行、漏一页均难以事后发现，通常等到客户查件才暴露。
- 关键前提（已确认）:
  - Tracking ↔ Order Reference 严格 1:1；面单「x2 两件装」高亮只针对产品本身，不影响单号关系。
  - 无货勾选页 0/1 为**逐订单**；SKU 级缺货由拣货员标在拣货单上，不进无货勾选页 → 扫码与现行纸质流程**信息量完全等价，无粒度损失**。
  - 清单在当日内封闭有效：拣货员手里只有早上打印那一份，滚动新增订单不进仓库；中途到货只是把原本要标无货的订单变回可发，该订单本就在清单内 → 白名单不会过期，无需「名单外但格式合法」的中间态。
  - stage2 有货/无货清单入口认 Order Reference 列。
- 意图: 扫码集合 = 有货集合。扫到 = 实发(0)，清单内未扫到 = 无货(1)。产出直接对齐 stage2 入口格式，把纸质勾选与手工转录两步一并消灭。
- 期望:
  - stage1 增产 `扫码清单{店}.csv`（Tracking / Order Reference / 序号 / 店），与拣货表+面单同源同批产出，零新增人工步骤。
  - 扫码端 = **单个 HTML 文件入库**，仓库机浏览器打开。零安装、零 venv、零码页问题（绕开启动器/打包那一整条维护链）；USB HID 与蓝牙 HID 无线枪对浏览器行为一致，换硬件零改动。
  - 校验三态（白名单封闭，无黄色档）：
    - 清单内 + 首次 → 绿底满屏大字 + 短促确认音 + 记录
    - 清单内 + 重复 → 红底 + 报警音 + 拒绝记录
    - 不在清单内（SF 号 / 订单号 / 非 LP 结构 / 名单外 LP）→ 红底 + 报警音 + 拒绝记录；提示需区分「不是 LP 格式」与「不在今日清单」，两者成因与处理不同
  - 全程无人工交互：不需点击确认，红色状态挂到下一次成功扫描为止。**声音为主异常通道**——400 单节奏下操作员手在贴单放箱，不会盯屏幕，视觉只负责「出问题后抬头一看知道是哪单」。
  - 界面常驻已扫计数与当前时间，供货代取件截止前判断进度。
  - 收工导出 `有货清单{店}.csv`（Order Reference + 0/1），直接投 GUI 阶段二入口。
- 风险（最高优先级，漏扫方向反转）: 现行「任一留空报警」护栏防的是漏返回导致误发；改扫码后未扫即默认无货，**不会留空，护栏自然失效**。实际已寄出但漏扫的包裹会被判成缺货 → 天猫不回传、客户查不到物流、并向缺货记录注入假数据污染补货判断，且看起来与真实数据无异，极难事后发现。护栏必须替换为**收工计数对账**：扫码计数 vs 交货代实际箱数 / 取件回单件数，对不上当场查，不延后。
- 待验证（动手前）: 无货勾选页多品订单是否为多行——merge_multiproduct 把 0/1 作订单级列纵向合并，若 stage2 逐行读且要求「多品订单全 0 才整单发货」，则导出需保留原行结构而非一订单一行，否则触发留空报警。回归方法：取一份真实无货勾选文件，用模拟扫码结果生成等价导出，两者分别喂 stage2，五张产出逐格比对。
- 上线方式: 前几日与纸质勾选并行，收工双向对账，重点测量漏扫率与警告音在仓库噪音下的可闻性；数据稳定后再撤纸质。
