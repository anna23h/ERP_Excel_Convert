# SPEC — VO 拉单 Excel 自动化（开发契约）

来源流程：Obsidian `VO拉单流程逻辑梳理.md`。本文件锁定脚本的输入/输出/规则，开发以此为准。

## 输入

| 名称 | 文件示例 | 关键列 |
|---|---|---|
| ERP 导出（可多份 VO/GW，concat） | `VO0612ERP原始文档233单.xlsx`（17列含 `External ID`） | `Order Reference`, `VO Tracking No`, `Order Lines/Product/Internal Reference`, `Order Lines/Product/Picking Name`, `Order Lines/Quantity`, `VO Delivery Type`, `Order Lines/Product/Barcode`, `Terms and conditions`, `Order Lines/Product/Quantity On Hand`, `External ID`；**订货用 3 个可选列**：`Order Lines/Product/FS`(供应商), `Order Lines/Product/Safety Stock`(安全库存), `Order Lines/Product/Supply Remark`(备注) |
| 完整天猫导出（**唯一天猫输入，必选**） | `0611天猫无筛选3w+订单.xlsx`（sheet `file`，全量、可跨多日） | `系统履约单号`, `履约单状态`, `面单申请状态`（**按列名取**） |
| 昨日发货表 | `1006发货表.xlsx`（sheet `GW`/`VO`） | `Order Reference`, `VO Tracking No` |

> **不再需要单独的「面单已完成名单」**：发货范围由脚本在完整天猫导出上跑二段式自动推出（见下）。
> 原因：面单已完成名单本就是完整天猫导出按 `面单已完成` 筛一刀的子集，可由代码重建；
> 而完整天猫导出还独家承担**负集分桶**（用 `履约单状态` 把发不了的单拆成 取消/无运单），无法被替代。
> 故二者中真正不可少的是完整天猫导出，已设为必选并合并为唯一天猫输入。

## 连接键
ERP `Order Reference` 取**后15位**（`right(,15)`，形如 `SCP...`）== 天猫 `系统履约单号`。
注意：极少数情况两个不同 `Order Reference` 后15位相同（连接键冲突），脚本需报出（可能误判状态）。

## 运行粒度（单店输入 × 两店天猫）
**工具按店各跑一次**：ERP 导出在 Odoo 端人为按店筛选，每次输入是**单店**（VO 或 GW）。
天猫后台无法按店筛，完整天猫导出含**两店**——经连接键 `∩ ERP`（单店）天然收敛到该店，
另一店的天猫行无 ERP 单匹配、自动忽略。故所有产出天然单店，**文件名带 `VO`/`GW` 后缀**。
内部仍保留「按 `Order Reference` 前缀拆店」逻辑作安全网（万一一次传两店也不串）。
注意：两店天猫混合后，`后15位` 连接键潜在碰撞面变大，保留碰撞报警。

## 发货范围（二段式，`step4_merge.done_keys_from_full`）
从完整天猫导出推「今天该发的订单」，**两段有序、不可颠倒**：
1. **履约闸门在前**：`履约单状态 ∈ {新订单, 商家已接单}`（= 发货前的活单），剔除 `已发货/履约完成(已收货)/履约取消/平台申请取消/发货后取消` 这些已发或已死的历史单。
2. **面单条件在后**：在活单里再留 `面单申请状态 = 已完成`（运单已就绪），排除 `待处理/生成中`。

顺序为何不能反：`面单已完成` 是粘性历史属性，三天前已发货的单它照样是「已完成」。若先按面单筛（或只按面单筛），会把已发走的老单重新捞回 → **重复发货**。必须先用 `履约单状态` 关时间/生命周期闸门，再用面单状态过滤就绪度。

## 分流（步骤4 核心，`step4_merge.classify4`）
按 ERP 订单 `_key` 打 `_cat`，优先级 取消 > 发货/已补运单 > 无运单：
- **取消**：`_key` ∈ 完整天猫导出中 `履约单状态 ∈ {履约取消, 平台申请取消}`。→ 回传ERP上传表(取消类)。
- **发货**：在上面二段式推出的发货范围里。→ 进拣货/面单。
- **已补运单**：发货 且 ERP `Terms and conditions` 已含 `无运单`（昨日无运单今日补出）。→ 仍发货；列入已补运单清单(去天猫打面单)；回传ERP上传表里剥掉 `无运单` 恢复原值；账单 D 同样剥。
- **无运单**：既不在发货范围也未取消。→ **从拣货/面单剔除**，回传ERP上传表(无运单类，Terms 前缀 `无运单`，已带则不重复加)。

另：**新订单**（`履约单状态 = 新订单` ∩ ERP）不进 classify4 四桶，单独输出「新订单获单清单」，
供员工复制系统履约单号去天猫后台批量获单（确认接单）。依赖完整天猫导出。

### 履约状态业务语义（运营视角，影响理解但不改判据）
- **新订单**：买家下单、商家完全没处理的状态；从下单即开始计时，到点「确认接单」前的时长被天猫考核（影响店铺评分）——这正是「新订单获单清单」存在的理由（导出单号→员工后台批量确认）。确认后变 `商家已接单`（仅状态变更，仓库未真正拣货）。
- **商家已接单 + 面单已完成**：是**异常/积压单**——今天之前就已接单、却至今没寄出（一旦寄出即变 `已发货`）。成因：①接单当天爆仓发不完，拖到今天；②缺货无法及时寄；③当时面单还 `待处理/未生成`，今天面单转 `已完成` 才可发（此条对应 ERP 侧的 `已补运单`）。
- 两种状态（新订单 / 商家已接单）+ 面单已完成都**照发**，进拣货+面单，不在产出里区分。

## 状态取值（天猫）
- `履约单状态`：`履约完成(已收货)` / `履约取消` / `平台申请取消` / `已发货` / `新订单` / `商家已接单` / `发货后取消(系统取消)`
- `面单申请状态`：`已完成` / `待处理` / `生成中`（判定发货的第二段闸门，见「发货范围」；脚本直接在完整天猫导出上按 `履约∈{新订单,商家已接单} ∧ 面单=已完成` 推出发货范围，不再依赖外部预筛名单）

## 输出表（A/B/C/D/E）

### A. 「拣货表+面单」workbook
- **拣货表** sheet（扁平单层，英文列名，一行一个 SKU）：
  `Internal Reference | Picking Name | Barcode | Quantity(求和) | Quantity On Hand(最大值)`
  实测每个 SKU 唯一对应一个 Picking Name/Barcode，扁平不丢数据；异常需报出。
  **按 `Internal Reference` 升序排序**（大小写不敏感、stable），便于捡货员按货号顺序拣货。此表行间独立(已聚合)，排序不影响数据；**面单/无货勾选 sheet 不可排序**（多品订单靠相邻行合并单元格、且两表靠 `序号` 逐行对应，排序会错位）。
- **面单** sheet（订单明细，6列 + 末尾空白「仓库备注」列）：
  `Order Reference | VO Tracking No | Internal Reference | Picking Name | Quantity | VO Delivery Type`
  只含**发货集合**（= classify4 的 发货+已补运单，已剔除无运单/取消）；出/无 拆分等仓库捡货后人工反馈。
  **按店铺(VO/GW)各出一份**（`YYYY年MM月DD日{ch}{n}单 拣货表+面单.xlsx`，序号店内独立 1..N）：仓库/打印按店进行。两店 ERP 可一次传入，自动分别出文件。
  标黄规则（满足任一）：
  1. `Internal Reference` 以 `x`+数字 结尾（多件装，正则 `x\d+$`，覆盖 x2/x3/x4/x10…，忽略大小写、结尾锚定）——提醒打包员该商品是多件装
  2. `Quantity` > 1
  3. `VO Delivery Type` == `CC`
  多品订单：订单级列 `Order Reference` / `VO Tracking No` / `VO Delivery Type` 纵向合并单元格（按列名定位，加 `序号` 列不影响）。
  注：实际列为 `序号` + 上述6列 + `仓库备注`，共8列（`序号` 与无货勾选 sheet 逐行对应）。
- **无货勾选** sheet（与面单**完全同版式**：同列、同顺序、同合并、同标黄，仅在最前面多一列 `无货(1=缺货)`）：
  - 由 `build_nogoods_helper` **复用 `build_facesheet`** 生成，与面单**数据同源**（同一 `facesheet`），故内容永远逐行一致；`无货(1=缺货)` 默认 0（0=有货，缺货填 1）。
  - 动机：仓库返回的纸质文档就是面单版式；屏幕表与纸面单一致，操作员对着纸面单逐行填 0/1 不会错位（旧版另起字段拼表、版式不同，已废弃）。
  - 阶段二 `classify_return` 按标注列（表头 `无货/缺货/勾选` 前缀）+ 行内 `SCP\d+` 键读取，与本表结构解耦，故改版式不影响判定。

#### 单元格格式（`build_excel.style_sheet` / `fix_merged_alignment`，三张表共用）
- **行高**：拣货表 / 面单 / 无货勾选（及其余辅助清单）统一 **35**（常量 `ROW_H`）。表头始终水平+垂直居中。
- **拣货表**：`Internal Reference` / `Picking Name` / `Barcode` 三列内容 **左对齐 + 垂直下沉(bottom)**；`Picking Name` 字号比其他**小 2 号**（15→13）。
- **面单 / 无货勾选**：`Order Reference` / `VO Tracking No` / `Internal Reference` / `Picking Name` 四列内容 **左对齐 + 垂直下沉(bottom)**；但该单元格若是**合并单元格** → 改 **左对齐 + 垂直居中**（`fix_merged_alignment` 在合并后覆盖首格）。其中 `Internal Reference` / `Picking Name` 字号**小 2 号**。
- 其余列（含 `Quantity` / `VO Delivery Type` 等）维持水平+垂直居中、15 号。

> **阶段二文件名统一约定**（`stage2.stage2_name`，对齐阶段一 `make_output_name`）：
> `YYYY年MM月DD日{渠道}{单数}单 {产出名}.xlsx`（`单数` 与 `产出名` 间留一个空格，日期=运行当天）。
> 适用 B/C/D/E。B/C/D 渠道取 `chan_suffix`（单店 `VO`/`GW`，混店 `VO+GW`，一份文件）；E 出库单逐店拆分，渠道为单店 `VO`/`GW`。**货代合并发货表(I)不在此约定内**，沿用固定前缀 `IHTCTGMBH+IH{YYYYMMDD}+{单数}.xlsx`。

### B. 系统履约单号表（上传天猫）
- 单列 `系统履约单号`（= Order Reference 后15位），来源 = 实际发货订单。
- 文件名：`YYYY年MM月DD日{渠道}{单数}单 系统履约单号.xlsx`（如 `2026年06月27日VO205单 系统履约单号.xlsx`）。

### C. 发货表
- 两个 sheet `GW` / `VO`，各2列：`Order Reference | VO Tracking No`。
- 文件名：`YYYY年MM月DD日{渠道}{单数}单 发货表.xlsx`（单数=两 sheet 合计发货订单数）。

### D. ERP 开账单上传表
- 文件名：`YYYY年MM月DD日{渠道}{单数}单 账单上传.xlsx`。
- `Order Date | ID | Order Reference | Terms and conditions`
- `Terms and conditions` = `账单MMDD` + 原值（渠道+运单）；去空列。
- `ID` = Odoo External ID，是导入更新订单的匹配键，无法凭空生成。
- **来源**：直接取订单导出(`--erp`)的 `External ID` 列生成（导 ERP 时勾上该列即可，一份文件搞定）。**GUI 不再提供账单模板入口**；CLI 仍保留 `--billing` 作为 ERP 无 `External ID` 时的回落，常态不需要。
- 去重到订单级（订单导出逐行，多品订单合一行，ID 取订单头行）。

### E. 出库单（回传 Odoo 标记出库）
- 输入：stock picking 全量导出「出库原始数据」（英文表头 7 列：`ID | Source Document | Reference | Creation Date | Carrier/ID | Tracking Reference | Status`）。`Source Document` 形如 `VO_TOF_SCP…`，可多份（VO/GW 各一份）。
- 过滤：`Source Document` 的 SCP ∈ **实际发货订单**（= 待全集 − 无货，与 B/C 同源 `get_shipped_orders`）。
- 改写：①把 `Tracking Reference` 整列覆盖成**统一发货日期** YYYYMMDD（默认运行当天）；②`Carrier/ID` 整列统一填**固定承运商外部 ID** `__export__.delivery_carrier_9_066799ca`（列不存在则新建）；其余列沿用 pool 原值，**英文表头、保留原 Status**（不改 Bereit/Ready）。空单元格写空串，不写字面 `nan`。
- 拆分：按 `Source Document` 前缀**逐店各一份**，文件名 `YYYY年MM月DD日{店铺}{单数}单 出库单.xlsx`（如 `2026年06月27日VO120单 出库单.xlsx` / `…GW…`；单数=该店出库行数）。
- 异常：对 pool 覆盖到的店铺，发货订单在 pool 里找不到对应 picking 的，单独列出 SCP 给用户（不静默）。

### F. 回传ERP销售上传表（步骤4.6/4.9，`build_excel`，按店各一份）
- **一张表三类合一**（替代旧的「取消单 + 无运单清单」两文件）：`回传ERP销售上传表VO.xlsx` / `…GW.xlsx`。
- schema：`Order Date | <External ID> | Order Reference | Terms and conditions`，去重到订单级（`External ID` 是导入匹配键）。
- 三类行各自的 `Terms`（ERP 端按关键词分别筛选后做对应动作）：
  - **取消**：`YYYY年MM月DD日平台订单取消`（覆盖）→ ERP 筛此词批量 Cancel。无完整天猫导出则无此类。
  - **无运单**：`无运单` + 原值（已带则不重复加）→ ERP 保留标记，下次过滤。
  - **已补运单**：原值（剥掉 `无运单`）→ ERP 解除标记，恢复可发。
- 三类都改了状态、都必须回传 ERP，否则修改无意义。无 `External ID` 列则无法生成（提示勾上）。

### G. 新订单获单清单（步骤4.4，`build_excel`，按店各一份）
- 单列 `系统履约单号`（= `履约单状态=新订单` ∩ ERP 的 `_key`），文件 `新订单获单清单VO.xlsx` / `…GW.xlsx`。
- **不按面单状态过滤**：含**全部新订单**——`面单已完成` + `面单待处理` 都要（与发货集/面单不同）。原因：面单待处理的新订单今天虽不打包（不进面单），但其单号仍需上传天猫**确认接单**（新订单→商家已接单），否则计时考核继续累计。
- 用途：员工复制履约单号 → 天猫后台批量获单（**确认接单**）。依赖完整天猫导出；没传则跳过。
- 注意：本表 = 新订单 ∩ **单店ERP**。若某新订单不在该店 ERP 导出里（如属另一店），由对应店那次运行覆盖；不是丢单。

### G2. 今日预计发货总获单清单（`build_excel`，按店各一份）
- 单列 `系统履约单号`（= 发货集 `_ship` 的 `_key`，即 `履约∈{新订单,商家已接单} ∧ 面单=已完成` ∩ ERP，**按单号去重，一单一行**），文件 `今日预计发货总获单清单VO.xlsx` / `…GW.xlsx`。
- 用途：今日**预计**发货那批订单的单号清单。**「预计」而非「真实」**——本表出在阶段一、口径是「面单已完成→今天打算发的单」；真实发货名单要等**阶段二**仓库返回「无货勾选」扣掉缺货后才确定：
  ```
  今日预计发货(阶段一)  −  阶段二无货勾选扣下的  =  真实发货(阶段二)
  ```
  （对应 `stage2`：实际发货 = 待全集 − 无货清单。）
- 与面单的关系：**同一批订单**，本表是面单覆盖的全部订单去重到「一单一行」的单号版；面单则展开到 SKU 行（多品订单多行）。
- 与 G 的区别：G(新订单获单清单)只含 `新订单`（含面单待处理）用于**确认接单**；本表是今日预计发货全集（新订单+商家已接单，**仅面单已完成**），用于覆盖整批去天猫后台批量获单。

### G3. 补货预判清单（`build_excel.build_reorder`，按店各一份）
- 用途：对接 `[[Solo 每日作战清单]]` 订货 SOP·**模式一 step 0 盘前预判**——拉单后即可扫出确定会短的下界，把"认产品+查价"挪到上午。
- 来源：整份 ERP 订单导出按 `Internal Reference` 聚合（**全部订单行，不限发货集**），`Quantity` 求和 = `今日需求`。
- 列：`产品名 | SKU | 今日需求 | 在售(On Hand) | 缺口(=需求−在售) | 安全库存 | 供应商(FS) | 红旗 | 备注`。
- 全部 SKU，**按缺口降序**（缺口>0 即确定会短）。`红旗`：备注含 `暂不采购/MHD/效期` 标 `⚠核实`（别自动下单，核实或问老板）。
- **依赖 ERP 含 FS/Safety Stock/Supply Remark 三列**，缺则跳过并提示（在 Odoo 订单导出模板勾上即可）。
- 两店 SKU 不重叠，故沿用按店拆分（Order Reference 前缀），无需跨店合并。文件 **`{店铺}补货预判清单.xlsx`**（`VO补货预判清单.xlsx` / `GW补货预判清单.xlsx`）。

### H. 已补运单清单（步骤4.8，`build_excel`，按店各一份）
- 单列 `系统履约单号`（= 已补运单订单的 `_key`），拿去天猫后台批量打面单。
- **按店各一份**（`已补运单清单VO.xlsx` / `…GW.xlsx`）：运单需分批次下载，避免捡货员混淆运单。

### I. 货代合并发货表（阶段二收尾，跨店，`stage2.build_forwarder`）
- **唯一跨店产出**：接受 **N 份发货表**（两店 × 当天多次拉单，多选作冗余），合并成一张给货代核对。
- 列：`Order Reference | Tracking number`（单 sheet `Sheet2`）。按 `Order Reference` **去重**；同单不同运单报冲突。
- 文件名**固定格式**：`IHTCTGMBH+IH{YYYYMMDD}+{发货单数}.xlsx`（只变日期与单数）。
- 入口模糊：自动定位 `Order Reference` 列（名或含 SCP）与 Tracking 列（名含 tracking/运单，或两列时取另一列）。

## 阶段二运行粒度与入口
阶段二(B/C/D/E + 缺货记录)**按店铺各跑一次**：单店 ERP + 该店返回文件。
因回传 ERP 取消/无运单/账单需按店区分、人工分别上传，脚本不做**跨店**合并以避免复杂度与不稳定。

**两个入口二选一**（都给则优先有货）：
- **有货清单**(白名单)：真实发货的履约单号/Order Reference，直接 ∩ ERP 取明细出 B/C/D/E。
  缺货记录 = 拣货候选(需 `--tmall-full`) − 有货（**整单级**，有货清单只到订单号，无法表达多品订单缺单件）。
- **无货勾选**(黑名单)：仓库返回标 0/1 的文件，**直接取有货(0)** 作发货集合（`classify_return`）。
  - 标记列(`无货/缺货/勾选`开头)：**1/真=无货，0=有货，空=未确认**。
  - 订单级裁决（多品订单取最保守）：**全部行 0 才整单发货**；任一行 1 → 整单不发(入缺货记录)；任一行空 → 整单不发 + **单独报警**。
  - 不再用「候选 − 无货」减法：漏返回某店/某单不会默认全发，更安全。`--tmall-full` 仅作交叉核对(发了候选外的单则报警)，非必需。
  - 缺货记录直接来自返回文件的 SKU 明细（**SKU 级**，多品订单可缺单件）。
    注：缺货记录已从阶段二移出（未来单独成阶段：无货清单×库存ERP 筛查），`build_shortage()`/`read_marked()` 暂保留休眠。二者仍按**旧列名** `SKU/商品名/数量/系统履约单号` 取数；而无货勾选 sheet 已改面单版式（列为 `Internal Reference/Quantity/Order Reference`、无 `商品名`），故复活该阶段时需把读取改成面单版列名（`商品名` 改从 ERP 按 SKU 补）。

入口文件**高度模糊**：不限 sheet 名/列名/列位置，按值模式 `SCP\d+` 识别连接键，其余列(数量/名称/日期)自动忽略；
均支持**多选作冗余**（同店分多次导出时合并，不用于 VO/GW 跨店合并）。

### 运单号取数（B/C/D 共用）
发货订单的 `VO Tracking No` 三级回落：**有货清单**(若传，`load_shipped_map` 按行配对 SCP↔运单) → **ERP** → **留空**。
有货清单是员工当场判断的最新结果，优先级高于 ERP 存量。

### 表级容错（绝不"完全没输出"）
阶段二各产出 **B/C/D/E 彼此独立**：每张表包在 `_step` 里，任一表缺数据或抛异常**只记错并跳过该表**，
不阻断其余产出。业务上员工宁可拿到部分结果 + 明确提示，也不要"一处出错全盘无输出"。典型跳过提示：
- 发货表：整批发货订单一个运单号都没有(有货清单和 ERP 都没给) → 跳过发货表，账单/出库照常。
- 出库单：未传 `--picking`，或出库原始数据无匹配的发货订单 → 跳过。
- 账单上传：订单导出无 `External ID` 列且未传账单模板 → 跳过(提示去 Odoo 勾上该列)。

阶段一加载侧同此原则：`step4_merge.load_erp` 只对**存在的**订单级列做 `ffill`，缺可选列不再崩在加载阶段。

## 为什么必须在脚本里筛/剔，而非只在天猫导出端筛（实测结论）
天猫导出端只能回答「谁有面单」，但拉单需要**发货/取消/无运单**三个桶，分桶依赖：① 与单店 ERP 批次取交集 ② ERP 的取消状态 ③ ERP 历史 Terms（已补运单）。这些信息分散在 ERP + 天猫全量，单次平台筛选表达不了。
实测（0611）：59 个「在 ERP、不在发货范围」的单，真实状态是 6 取消 + 53 无运单**混在一起**——只看「在不在发货范围」一律排除，分不出该作废还是该暂扣。故必须用完整天猫导出的 `履约单状态` 在负集上做分桶。
完整性保证：classify4 四桶穷尽互斥，每单必落且仅落一桶（实测 310 = 246+5+53+6，无遗漏）。

## 全局规则
- 标签统一英文优先（`账单MMDD`，废弃 `开发票` 旧叫法）。
- 取消两状态合并：`履约取消`/`平台申请取消` 同归「取消」，统一打 `平台订单取消`。`发货后取消(系统取消)` 暂不处理（用户定）。
- 去冗余：统一拣货表格式、删空列、删重复列（如 `履约单号(文本)`）。
- 跨多个文件选格式时**先问用户**再定。
- 异常（delivery type 为空、SKU 多 Barcode、连接键后15位冲突）一律单独列出，不静默丢弃。
- **产出重名不覆盖**：所有保存点经 `unique_path()`，目标文件已存在则自动加 `(1)/(2)…` 后缀，绝不覆盖旧产出（一天多次拉单、同名产出可共存对照）。
- **护栏**：传了完整天猫导出时，对发货集合反查其真实状态——若是 `已发货/履约完成(已收货)/发货后取消`（不该出现在今日名单），报警提示名单可能过期、当心重复发货。
