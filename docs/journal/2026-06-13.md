# 2026-06-13（下午）业务流程重新对齐 + 七项改动

新接手者把对业务的进一步理解同步了一遍，纠正了多处此前的偏差，据此改代码。

## 改了什么
1. **新订单获单清单**（缺口补全）：`step4_merge` 加 `NEW_ORDER_STATUS`；`build_excel.build()` 从完整天猫导出取 `履约单状态=新订单` ∩ ERP → 单列系统履约单号 `新订单获单清单{ch}.xlsx`，供员工去天猫批量获单。
2. **货代合并发货表**（跨店唯一产出）：`stage2.build_forwarder` 吃 N 份发货表，去重(按 Order Reference)，列 `Order Reference | Tracking number`，固定命名 `IHTCTGMBH+IH{YYYYMMDD}+{单数}.xlsx`；CLI `--forwarder`、GUI 独立入口。
3. **上传表三合一**：删 `取消单.xlsx`/`无运单清单.xlsx`，合并为 `回传ERP销售上传表{ch}.xlsx`，含取消/无运单/已补运单三类 Terms 写回；已补运单在 `build()` 里也算"剥无运单恢复原值"。
4. **阶段二黑名单改"直接取有货(0)"**：`stage2.classify_return` 按 0/1/空裁决，多品订单**全 0 才整单发**，任一 1 不发(入缺货)，任一空不发+报警；替换"候选−无货"减法，消除"漏返回某店默认全发"。

## 决策（这次校准出来的关键业务理解）
- **履约单状态 / 面单状态是两个正交维度**，处理时先筛履约单(新订单获单/取消标记)、再筛面单(待处理→无运单写回)。此前我把"面单已完成名单"误当单一筛选器。
- **运行按店**：ERP 单店导出、天猫两店混合，经 `∩ERP` 收敛；所有产出带 VO/GW 后缀，暂不跨店合并(便于人工核对)；拆店逻辑留作安全网。
- **三类都必须回传 ERP**（都改了状态），故合并进一张上传表。
- **新订单不写"商家已接单"回 ERP**：该状态在天猫侧获单产生、下次拉单同步；员工视频实证上传仅 4 列、不含履约单状态。
- 货代表**留冗余**：一天可能 2–3 次拉单 → N 份发货表多选合并。
- 护栏维持反查完整天猫(用户认为更准)；两个盲区(ERP有天猫无→静默无运单 / 发货后取消)靠人工同时下载 + 既有决定，代码不改。
